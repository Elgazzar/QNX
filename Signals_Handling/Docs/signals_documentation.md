# Signal Handling Documentation

This document describes the signal handling APIs used in the `Signals_Handling.cpp` codebase and provides an overview of the common signals available in the QNX RTOS environment.

## Signal Handling APIs Used

### 1. `sigemptyset`
**Signature:** `int sigemptyset(sigset_t *set);`
**Description:** Initializes a signal set to be completely empty. This means no signals are currently included in the set. It is standard practice to clear a set before explicitly adding signals to it.

### 2. `sigaddset`
**Signature:** `int sigaddset(sigset_t *set, int signo);`
**Description:** Adds a specific signal (like `SIGINT` or `SIGTERM`) to the initialized signal set.

### 3. `sigfillset`
**Signature:** `int sigfillset(sigset_t *set);`
**Description:** Initializes a signal set to contain *all* available system signals. Note that even if all signals are added, OS-level uncatchable signals (like `SIGKILL` and `SIGSTOP`) will still bypass user handling.

### 4. `pthread_sigmask`
**Signature:** `int pthread_sigmask(int how, const sigset_t *set, sigset_t *oldset);`
**Description:** Examines or changes the signal mask of the calling thread. In multithreaded applications, it is considered best practice to block signals in all threads (using the `SIG_BLOCK` flag) and then dedicate a specific thread to process them. This prevents signals from interrupting your normal thread execution flow.

### 5. `sigwait`
**Signature:** `int sigwait(const sigset_t *set, int *sig);`
**Description:** Suspends the execution of the calling thread until one of the signals specified in `set` becomes pending. Once received, `sigwait` removes the signal from the pending list and returns the signal number in the `sig` pointer. This is a highly recommended synchronous approach to signal handling in multi-threaded programs (as opposed to using traditional asynchronous `signal()` or `sigaction()` handler functions).

### 6. `strsignal`
**Signature:** `char *strsignal(int sig);`
**Description:** Takes a signal number and returns a human-readable string describing the signal (e.g., "Interrupt" for `SIGINT`). This is very useful for logging and debugging purposes.

### 7. `pthread_join`
**Signature:** `int pthread_join(pthread_t thread, void **value_ptr);`
**Description:** Blocks the calling thread (the main thread in your case) until the specified target thread terminates.

## Signals in QNX

QNX Neutrino RTOS is highly POSIX compliant and supports both standard POSIX signals and Real-Time POSIX signals.

### Standard POSIX Signals
These are the standard signals you might send via the command line (using `kill -<SIGNAL> <PID>`) or programmatically:

| Signal | Value (Typical) | Description |
| :--- | :--- | :--- |
| **`SIGINT`** | 2 | **Interrupt**: Sent when the user presses `Ctrl+C` in the terminal. |
| **`SIGQUIT`** | 3 | **Quit**: Sent when the user presses `Ctrl+\`. |
| **`SIGILL`** | 4 | **Illegal Instruction**: The process attempted to execute a malformed instruction. |
| **`SIGABRT`** | 6 | **Abort**: Triggered by the `abort()` function (often used by `assert()` failures). |
| **`SIGFPE`** | 8 | **Floating-Point Exception**: Erroneous arithmetic operation (e.g., divide by zero). |
| **`SIGKILL`** | 9 | **Kill**: Immediately terminates the process. **This signal cannot be caught, blocked, or ignored.** |
| **`SIGSEGV`** | 11 | **Segmentation Fault**: Invalid memory reference (e.g., dereferencing a null or invalid pointer). |
| **`SIGALRM`** | 14 | **Alarm**: A timer signal generated by the `alarm()` function. |
| **`SIGTERM`** | 15 | **Termination**: A polite request to terminate the process. This is the default signal sent by the `kill` command. Apps should catch this to clean up before exiting. |
| **`SIGUSR1`** | 16 | **User Defined 1**: Available for custom application logic. |
| **`SIGUSR2`** | 17 | **User Defined 2**: Available for custom application logic. |
| **`SIGCHLD`** | 20 | **Child Status Changed**: Sent to a parent process when its child stops or terminates. |

### POSIX Real-Time Signals (`SIGRTMIN` to `SIGRTMAX`)
QNX heavily relies on POSIX Real-Time signals (typically signals ranging from 41 to 56, defined between `SIGRTMIN` and `SIGRTMAX`). 

Unlike standard signals:
1. **Queuing:** If multiple instances of a real-time signal are sent to a process, they are queued. Standard signals are usually compressed (if you send `SIGINT` 5 times before it's processed, the handler might only see it once).
2. **Payloads:** Real-time signals can carry accompanying data (an integer or a pointer) via the `siginfo_t` structure.

In QNX, Real-Time signals are frequently used by the OS for:
*   Timer expirations (`timer_create`).
*   Asynchronous I/O notifications (`aio_*` functions).
*   Message queue notifications (`mq_notify`).
